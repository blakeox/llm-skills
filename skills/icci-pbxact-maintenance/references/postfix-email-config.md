# Postfix Email Configuration — AWS SES Relay

## Overview

All PBXact instances relay outbound email through AWS SES (Simple Email Service)
via Postfix SMTP AUTH. This handles voicemail notifications, call recording
delivery, system alerts, and any other email generated by FreePBX.

---

## Required Package

```bash
apt-get install -y libsasl2-modules
```

This is the #1 most common email failure on fresh Debian installs. Without it,
Postfix cannot authenticate to AWS SES and you get:

```
SASL authentication failed; cannot authenticate to server
email-smtp.us-east-1.amazonaws.com: no mechanism available
```

This package is now pre-installed in the golden master image.

---

## Postfix main.cf Configuration

The following lines are added to `/etc/postfix/main.cf` for AWS SES relay:

```ini
# SMTP relay through AWS SES
relayhost = [email-smtp.us-east-1.amazonaws.com]:587
smtp_sasl_auth_enable = yes
smtp_sasl_security_options = noanonymous
smtp_sasl_password_maps = hash:/etc/postfix/sasl_passwd
smtp_tls_security_level = encrypt
smtp_tls_note_starttls_offer = yes

# SMTP Generic maps — rewrites sender addresses
smtp_generic_maps = hash:/etc/postfix/generic

# Force IPv4 (AWS SES works better with IPv4)
smtp_address_preference = ipv4
inet_protocols = ipv4

# Message size limit (24MB — accommodates call recording attachments)
message_size_limit = 24576000
```

## SASL Password File

Create `/etc/postfix/sasl_passwd`:

```
[email-smtp.us-east-1.amazonaws.com]:587 AKIAXXXXXXXXXXXXXXXX:XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
```

The credentials are AWS SES SMTP credentials (not IAM access keys — SES generates
its own SMTP credentials from IAM users).

```bash
postmap /etc/postfix/sasl_passwd
chmod 600 /etc/postfix/sasl_passwd /etc/postfix/sasl_passwd.db
```

---

## Generic Maps — Sender Address Rewriting

PBXact generates email from internal hostnames (freepbx.localdomain, localhost).
These must be rewritten to the client's domain or SES will reject them.

Create `/etc/postfix/generic`:

```
# Rewrite all internal sender addresses to the PBX's domain
@freepbx.localdomain          @talk.<clientdomain>.com
@localhost                     @talk.<clientdomain>.com
root                           pbx@talk.<clientdomain>.com
asterisk                       pbx@talk.<clientdomain>.com
root@localhost                 pbx@talk.<clientdomain>.com
root@localhost.localdomain     pbx@talk.<clientdomain>.com
root@freepbx                   pbx@talk.<clientdomain>.com
root@freepbx.localdomain       pbx@talk.<clientdomain>.com
asterisk@localhost             pbx@talk.<clientdomain>.com
asterisk@localhost.localdomain pbx@talk.<clientdomain>.com
asterisk@freepbx              pbx@talk.<clientdomain>.com
vm@asterisk                    pbx@talk.<clientdomain>.com
```

After editing:
```bash
postmap /etc/postfix/generic
postfix reload
```

## Testing Email

```bash
echo "Test email from PBX" | mail -s "SMTP Test" test@example.com
tail -f /var/log/mail.log
```

**Successful delivery looks like:**
```
status=sent (250 Ok <message-id>)
```

**Common failures:**

| Log message | Cause | Fix |
|-------------|-------|-----|
| `SASL authentication failed; no mechanism available` | Missing libsasl2-modules | `apt install -y libsasl2-modules && systemctl restart postfix` |
| `relay access denied` | sasl_passwd not hashed or wrong credentials | `postmap /etc/postfix/sasl_passwd` and verify SES credentials |
| `message rejected: Email address is not verified` | SES sandbox or unverified sender | Verify domain in SES console or request production access |
| `Connection timed out` | Security group blocking outbound 587 | Add outbound rule for TCP 587 |

## Verifying Email Headers

After a successful test email, check the received message headers for:

- **DKIM pass** on the client domain — SES signs with the domain's DKIM key
- **DKIM pass** on amazonses.com — SES's own signature
- **SPF pass** — SES is an authorized sender
- **From:** should show the rewritten address (e.g., `pbx@talk.clientdomain.com`)

---

## OpenDKIM — Do NOT Use

OpenDKIM is redundant when relaying through AWS SES. SES handles all DKIM signing
at the relay level. Adding OpenDKIM on the PBX:
- Adds complexity for no benefit
- Can cause double-signing issues
- Is one more service to maintain

If OpenDKIM is installed from a previous configuration:

```bash
systemctl stop opendkim
systemctl disable opendkim
```

Remove any milter lines from main.cf if present:
```ini
# REMOVE these if present (should already be commented out):
# milter_default_action = accept
# milter_protocol = 6
# smtpd_milters = unix:/opendkim/opendkim.sock
# non_smtpd_milters = unix:/opendkim/opendkim.sock
```

## Per-Client Setup Procedure

When setting up email on a new client's PBXact instance:

1. Ensure `libsasl2-modules` is installed (should be from GM)
2. Configure `/etc/postfix/sasl_passwd` with SES SMTP credentials
3. Run `postmap /etc/postfix/sasl_passwd && chmod 600 /etc/postfix/sasl_passwd*`
4. Configure `/etc/postfix/generic` with client's domain
5. Run `postmap /etc/postfix/generic`
6. Add relay and generic map lines to main.cf
7. Run `postfix reload`
8. Send test email and verify in `/var/log/mail.log`
9. Disable opendkim if present
10. Verify email headers show DKIM pass and correct sender
